# Отчёт по автоматизированному визуальному тестированию веб-приложения

**Объект тестирования:** главная страница Google с **русским** интерфейсом — `https://www.google.com/?hl=ru&gl=RU`, в Chrome заданы язык `ru-RU` и `Accept-Language: ru`.

---

## 1. Цель работы

Настроить среду автоматизированного тестирования (Selenium WebDriver, pytest), реализовать сценарии со **скриншотами** (полная страница и отдельный элемент интерфейса), подключить **Allure** для сбора результатов и вложений, провести прогон и оценить качество отображения страницы по полученным данным.

---

## 2. Ключевые элементы интерфейса для скриншотного тестирования

| Элемент | Способ захвата |
|--------|----------------|
| Вся видимая страница (viewport) | `driver.get_screenshot_as_file(...)` |
| Поле поиска (форма ввода запроса) | `element.screenshot(...)` для `textarea/input[name='q']` |

---

## 3. Тестовое окружение

| Компонент | Версия / примечание |
|-----------|---------------------|
| ОС | macOS |
| Язык | Python 3 |
| Фреймворк тестов | pytest |
| Автоматизация браузера | Selenium 4, **Google Chrome** (режим `--headless=new`) |
| Сравнение изображений | Pillow, библиотека **ImageHash** (perceptual hash `phash`) |
| Отчётность | **Allure** (`allure-pytest`), каталог результатов `allure-results/` |

Дополнительные настройки Chrome: окно **1920×1080**, язык интерфейса **русский** (`--lang=ru-RU`, предпочтения языка в `prefs`), `--force-device-scale-factor=1`, скрытие полос прокрутки.

---

## 4. Описание автоматизированных тестов

### 4.1. Регрессия внешнего вида полной страницы (`test_google_ui_fullpage`)

1. Открывается страница Google с локалью в URL (`hl=ru`, `gl=RU`).
2. При появлении экрана согласия на cookies (в т.ч. «Прежде чем перейти к Google») выполняется клик по **«Принять все»** / **Accept all** или при необходимости **«Отклонить все»** — в том числе внутри `iframe`, пока не станет видна главная со строкой поиска.
3. Ожидается `document.readyState === "complete"` и **видимость** поля поиска (`textarea`/`input[name='q']`).
4. Сохраняется скриншот в `diffs/google_current.png`.
5. **Первый запуск:** если файла эталона `baselines/google.png` нет, текущий скриншот переименовывается в эталон, тест помечается как пропущенный (`skip`) с пояснением — нужен повторный запуск для сравнения.
6. **Последующие запуски:** вычисляются perceptual hash (`phash`) для эталона и текущего изображения. Если разница хешей **больше 5**, тест считается проваленным: строится карта отличий (`ImageChops.difference`) и сохраняется в `diffs/google_diff.png`.

Порог `5` допускает небольшие визуальные отличия (антииалиасинг, мелкие анимации), но отлавливает заметные изменения макета.

### 4.2. Скриншот элемента — поле поиска (`test_google_search_form_element`)

Используется тот же переход на Google с обработкой окна согласия, что в п. 4.1. Затем находится поле ввода (`textarea` или `input` с `name='q'`), делается скриншот только этого элемента, файл: `diffs/google_search_form.png`. Тест проверяет, что файл успешно создан.

### 4.3. Интеграция с Allure

К скриншотам и текстовым вложениям (значение разницы хешей) применяется `allure.attach` / `allure.attach.file`, шаги оформлены через `@allure.step`, сценарии — через `@allure.feature`, `@allure.story`, `@allure.title`.

## 5. Запуск тестов и формирование отчёта Allure

Из корня проекта (один раз: `python3 -m venv .venv`, активация venv, `pip install -r requirements.txt`):

```bash
pytest
allure generate allure-results -o allure-report --clean
python tools/build_report_html.py
```

Просмотр HTML-отчёта: открыть файл `allure-report/index.html` в браузере или выполнить `allure open allure-report`. В карточке каждого теста в разделе вложений отображаются прикреплённые PNG.

## 7. Анализ и выводы

1. **Корректность отображения:** на скриншотах видна стандартная главная страница Google: логотип, поле поиска, кнопки сервисов. Критичных визуальных дефектов (поломка вёрстки, отсутствие ключевых блоков) по сохранённым изображениям не наблюдается.

2. **Стабильность сценария:** в URL явно заданы `hl=ru` и `gl=RU`, в Chrome — русский язык интерфейса; ожидание видимости строки поиска перед скриншотом даёт предсказуемую «целую» страницу. Эталонное сравнение по `phash` отмечает существенные изменения макета при повторных запусках.

3. **Ограничения:** внешний вид может меняться со стороны Google (A/B-тесты, баннеры, праздничные логотипы) — в таких случаях эталон следует обновить (удалить `baselines/google.png` и заново пройти цикл «первый запуск»).

4. **Allure:** отчёт объединяет статусы тестов, пошаговое выполнение и вложения-скриншоты, что удовлетворяет требованию зафиксировать ход проверки и визуальные доказательства.

## 8. Заключение

Выполнены настройка среды (Selenium + Chrome + pytest + Allure), разработка скриншотных тестов для полной страницы и для элемента интерфейса, проведён прогон с сохранением изображений и формированием отчёта Allure. По полученным скриншотам и логике сравнения с эталоном можно судить о стабильности отображения выбранной публичной страницы при регрессионном тестировании.
